### 实现笔记（Phase 1）

**根因修正**：原 `buildSuggestionItems` 用 `prompt.split(/\s/).at(-1)` 取**整段 prompt** 最后一个空白分词，完全忽略光标。光标停在文字中间时，末尾分词在光标之后 → 文件/skill 菜单弹不出。这才是用户「只能行首引用」的真正原因，而非行首判断。

**触发规则**（`detectTrigger`）：
- 取光标前最后一个 `@` 或 `/`，要求从该字符到光标连续无空白、无其它触发符。
- `/` 命令：前一字符必须是空白/起始/`(` `[`，避免路径 `a/b`、URL `https://` 误触。
- `@` 引用：前一字符若是字母/数字则视为 `email@host` 不触发；其它字符（含中日韩、标点、空白）均允许 → 满足中文写作「写一段话@文件」不打空格的习惯。

**待验证（人工）**：
- 中文输入法下连续输入「话@文件」能否弹菜单（IME composition 期间 `onChange` 时机）。
- 多行文本中间行输入 `@` / `/`。
- 选中后光标定位、继续输入不重复触发（已插入项含 `/` 与尾部空格，`detectTrigger` 会因 segment 含 `/`/空白返回 null，已规避）。

### 实现笔记（Phase 2）

**关键设计决策**：单一数据源仍是外部 `value`（字符串），chip 以 `@path` / `/command` 内联文本存在于字符串中。contentEditable 只是受控渲染层——chip 是 `contenteditable=false` 的 span，其余为文本节点。这样 `sendPrompt` / shell 模式 / 命令历史 / `detectTrigger` / `applySuggestion` 全部沿用纯文本偏移逻辑，**无需改后端协议**。

**光标偏移统一为纯文本偏移**：chip 贡献其 `data-raw` 字符数，与 textarea 的 `selectionStart` 语义一致，保证 Phase 1 的光标相关触发检测可直接复用。

**三个易错点处理**：
1. **IME 中文**：`compositionstart/end` 期间 `composingRef` 锁定，不回写 value、不触发 onChange，避免 composition 中途被 React 重渲染打断丢字。
2. **受控回写冲突**：重渲染时机从「DOM 纯文本 ≠ value」改为「chip 区间集合变化」。光标在某个 token 内部编辑时（如 `@src|`）不 chip 化，保持纯文本可继续输入；光标移出 token 后才 chip 化。避免用户输到一半被 chip 化导致光标被推出。
3. **粘贴富文本**：拦截 paste，只取纯文本 `insertText`，防止外部样式污染 contentEditable。

**换行策略**：contentEditable 默认 Enter 插 `<div>/<br>` 会破坏扁平文本模型。RichInput 在上层未 `preventDefault` 的 Enter（即换行场景）手动 `insertText('\n')` + `white-space:pre-wrap` 渲染，DOM 始终保持「文本节点 + chip」。

**shell 模式适配**：`.composer-box.shell-mode .rich-input` 加 `padding-left:18px` 与 `::before` 指示条对齐；`.bang`/`.bang-bang` 的 `caret-color` 同步到 `.rich-input`。

**`caretRef` 机制**：App 用 `pendingComposerCaretRef` 设置目标光标偏移，传给 RichInput 的 `caretRef` prop。RichInput 重渲染时优先读 `caretRef.current` 恢复光标，避免 rAF 竞态导致光标跳错位置。

**组件结构**：`RichInput` 用 `forwardRef` 暴露根 div，App 通过 `composerTextareaRef`（已改为 `HTMLDivElement`）拿到 contentEditable 根，用于光标读取/恢复和 `ensureComposerTailVisible`/`syncComposerAutoHeight` 的 `scrollHeight` 计算。
