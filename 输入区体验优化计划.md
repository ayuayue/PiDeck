## 问题分析

### 当前实现逻辑（InputArea.tsx）

1. **触发条件限制**：`handleInputChange` 中检测 `@` 和 `/` 时使用了行首判断逻辑：
   - `@` 菜单：检测当前行是否以 `@` 开头（`lineBeforeCursor` 匹配 `/@(\S*)$/`）
   - `/` 菜单：检测整行输入是否以 `/` 开头（`trimmedValue.startsWith('/')` 或类似逻辑）
   
   这导致只有在新行开头或输入框为空时才能唤起菜单。

2. **渲染方式**：选中的文件/skill 被直接插入为纯文本（如 `@filename` 或 `/skill:name`），在视觉上与普通文字无区别。

### 用户期望

- 在文字中间任意位置输入 `@` 或 `/` 都能唤起对应菜单
- 选中后以卡片/chip 形式渲染，与普通文本有明显视觉区分

## 方案一：行内触发 @ 和 / 菜单

### 核心改动

将触发检测逻辑从"行首匹配"改为"光标前最近的触发符匹配"。

#### 改动文件：`src/renderer/src/components/input/InputArea.tsx`

**当前逻辑（约 L220-280 handleInputChange）：**
- 检查 `lineBeforeCursor` 或整行 `startsWith`
- 只有行首/输入框首字符为 `@` 或 `/` 才进入菜单状态

**改为：**

```typescript
// 从光标位置向前查找最近的触发字符
// @ 触发：光标前文本匹配 /(^|\s)@(\S*)$/
// / 触发：光标前文本匹配 /(^|\s)\/(\S*)$/
const textBeforeCursor = value.substring(0, cursorPosition)

// @ 文件/上下文引用
const mentionMatch = textBeforeCursor.match(/(^|[\s(])@([^\s@]*)$/)
if (mentionMatch) {
  setMentionQuery(mentionMatch[2])
  setShowMentionMenu(true)
  setMentionTriggerIndex(textBeforeCursor.length - mentionMatch[2].length - 1)
} else {
  setShowMentionMenu(false)
}

// / skill/命令调用
const slashMatch = textBeforeCursor.match(/(^|[\s(])\/([^\s/]*)$/)
if (slashMatch) {
  setSlashQuery(slashMatch[2])
  setShowSlashMenu(true)
  setSlashTriggerIndex(textBeforeCursor.length - slashMatch[2].length - 1)
} else {
  setShowSlashMenu(false)
}
```

**关键点：**
- 触发条件改为：`@` 或 `/` 出现在空格后、行首、或括号后即视为触发
- 新增 `mentionTriggerIndex` / `slashTriggerIndex` 状态，记录触发符在文本中的位置
- 菜单定位改为相对于光标位置浮动（需计算光标坐标）

#### 光标位置计算

需要新增一个工具函数来获取 textarea 中光标的像素坐标，用于弹出菜单定位：

```typescript
// 使用 mirror div 技术计算 textarea 中光标的屏幕坐标
function getCaretCoordinates(textarea: HTMLTextAreaElement, position: number): { top: number; left: number } {
  const mirror = document.createElement('div')
  const style = window.getComputedStyle(textarea)
  // 复制 textarea 样式到 mirror
  ;['font', 'padding', 'border', 'boxSizing', 'width', 'lineHeight', 'letterSpacing', 'wordSpacing'].forEach(prop => {
    mirror.style[prop] = style[prop]
  })
  mirror.style.position = 'absolute'
  mirror.style.visibility = 'hidden'
  mirror.style.whiteSpace = 'pre-wrap'
  mirror.style.wordWrap = 'break-word'
  
  const text = textarea.value.substring(0, position)
  mirror.textContent = text
  const span = document.createElement('span')
  span.textContent = textarea.value.substring(position) || '.'
  mirror.appendChild(span)
  document.body.appendChild(mirror)
  
  const coordinates = {
    top: span.offsetTop - textarea.scrollTop,
    left: span.offsetLeft
  }
  document.body.removeChild(mirror)
  return coordinates
}
```

#### 菜单定位改动

`MentionMenu` 和 `SlashMenu` 当前使用固定定位（相对输入框上方）。需要改为：
- 接收 `anchorPosition: { top: number; left: number }` prop
- 菜单渲染在光标上方或下方（根据空间自动判断）

## 方案二：引用项渲染为卡片样式

### 核心思路

将 textarea 替换为 **contentEditable div**（或使用带有 inline-decoration 的自定义编辑器方案），使得选中的引用项可以渲染为 inline chip 节点。

### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| A. 保留 textarea + 上方展示 chip | 简单稳定，不动核心输入逻辑 | chip 和文本分离，不在上下文中 |
| B. 替换为 contentEditable div | 引用项直接在行内显示为 chip | 改动大，需处理光标、IME、撤销等 |
| C. 混合方案：textarea 保持不变，视觉层覆盖 overlay | 中等复杂度 | overlay 与 textarea 同步有挑战 |

### 推荐方案：B — contentEditable div

理由：
- 这是业界主流做法（Notion、Slack、Discord、飞书均用此方案）
- 一次改动解决两个问题（行内触发 + 卡片渲染）
- 长期可维护性好

#### 4. 数据提取

发送消息时，从 contentEditable div 中解析出：
- 纯文本内容
- 引用的文件列表（从 `data-type="mention"` 节点提取）
- 引用的 skill 列表（从 `data-type="skill"` 节点提取）

```typescript
function extractInputData(editorEl: HTMLDivElement) {
  const mentions: string[] = []
  const skills: string[] = []
  
  editorEl.querySelectorAll('.input-chip').forEach(chip => {
    const type = chip.getAttribute('data-type')
    const id = chip.getAttribute('data-id')
    if (type === 'mention' && id) mentions.push(id)
    if (type === 'skill' && id) skills.push(id)
  })
  
  // 获取纯文本（chip 替换为其标识符）
  const text = serializeToPlainText(editorEl)
  
  return { text, mentions, skills }
}
```

## 技术风险与备选方案

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| contentEditable 中文 IME 兼容性 | 中文输入时光标跳动或丢字 | 使用 `compositionstart` / `compositionend` 事件锁定，期间不触发菜单 |
| contentEditable 撤销栈丢失 | Cmd+Z 行为不符合预期 | 自行维护操作历史栈，或使用 `document.execCommand` 保持浏览器原生撤销 |
| 菜单定位在多行文本中偏移 | 弹窗出现在错误位置 | 使用 `window.getSelection().getRangeAt(0).getBoundingClientRect()` 精确定位 |
| 粘贴富文本导致样式污染 | 从外部粘贴带格式内容 | 拦截 paste 事件，强制转为纯文本插入 |
| 发送时数据解析复杂 | chip 节点和文本混合，需正确序列化 | 定义明确的序列化规则，chip → `@{id}` 或 `/{skill}`，写单元测试覆盖 |

### 降级方案

如果第二阶段 contentEditable 复杂度过高，可采用折中方案：
- 保持 textarea 输入
- 选中 @ 或 / 后，在 textarea **上方**以 chip 行展示已引用的文件/skill
- textarea 中仍显示纯文本标记（如 `@main.ts`），但用不同颜色高亮（通过透明 overlay 层）

## 预期效果

改造完成后：
1. 用户在任意位置输入 `@` 即弹出文件选择菜单，输入 `/` 即弹出 skill 菜单
2. 选中后在输入框中显示为带颜色的卡片 chip，文件为蓝色，skill 为紫色
3. 卡片可 hover 查看详情，点击 × 可删除
4. 发送时正确提取引用信息传递给后端
